WHY ARE WE BURNING MORE TOKENS THAN BASELINE?
==============================================================================

NOTE: la DB fue purgada entre eval; no hay prompt_tokens del baseline.
Solo se compara stop_reason + wallclock; el resto son numeros absolutos
del post batch + razonamiento causal.

Factor 1 - Mas runs terminan en stopped_by_budget:
  baseline:  4 judge_confirmed / 4 stopped_by_budget
  post:      3 judge_confirmed / 5 stopped_by_budget
  -> cada run que YA NO termina en judge_confirmed agota el budget
     (tipicamente 3-6x mas LLM calls que un judge_confirmed temprano).
  -> NO es regresion de A1+A2: es Phase 1 (round-aware threshold,
     stuck_planner, academic cascade) decidiendo que necesita mas evidencia.

  Runs que perdieron judge_confirmed: ['Q5']
  -> Estos runs hoy hacen budget completo en vez de aceptar la primera ronda.
     Si su confianza final >= 0.7, la calidad se mantiene; solo cuesta mas.

Factor 2 - Volumen por run (post batch, promedio):
  avg LLM calls/run:        20.8
  avg planning_rounds/run:  0.0
  avg reformulations/run:   0.0
  avg prompt_tokens/run:    17,317
  cost total post batch:    $1.0899
  -> Cada round extra dispara >=1 PLANNER + >=1 JUDGE + >=1 SYNTHESIZER call.

Factor 3 - CLASSIFIER cache (A2):
  CLASSIFIER calls registrados:  17
  Esperado sin cache (1/run):    8
  Tokens CLASSIFIER:             14,336 pt / $0.0205
  -> El cache short-circuit emite el evento 'classifier_cache_hit' en logs
     pero NO emite CostIncurred. Si el numero esta cerca de 8, no hubo
     repeticiones (cada Q es unica). El valor de A2 se vera en demos donde
     se repite la misma pregunta.

Factor 4 - Anthropic prompt cache (A1):
  CostIncurred NO desagrega cache_read_input_tokens vs cache_creation_input_tokens.
  Litellm los reporta en usage.* pero el payload del evento solo guarda
  prompt_tokens (suma total).
  -> cost_usd YA refleja el descuento (cache_read se factura ~10%) pero el
     conteo de tokens no distingue. Para validar A1 directamente:
       ssh oracle 'journalctl -u novum.service -n 2000 | grep -i cache_read'
  -> Commit pendiente: exponer cache_read / cache_creation en CostIncurredEvent.

Resumen ejecutivo:
  - Subida de tokens = mayormente Phase 1 (1 runs
    adicionales agotando budget x ~3-6x calls/run).
  - A1+A2 bajan costo POR CALL en Anthropic (visible en cost_usd, no en
    prompt_tokens) pero no anulan el volumen extra de Phase 1.
  - Proximos pasos: commit #2 (B2 regression harness) y commit #3 (B1 BM25+MMR)
    para reducir contexto por call y compensar el volumen extra.
